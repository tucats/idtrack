# PROMPTS

This document tracks the prompts to Claude Code used to create the idtrack application.

## PROMPT 1

Let's create an issue tracking system called IDTRACK. It will be a simple javascript/html/css web app and a back-end server
that provides authentication, database support, and other functionality. The backend will be written in Go.

### Directory components

I have provided the following elements to help get started.

- This file, PROMPTS.md which will contain a record of the primary prompts to Claude Code
- main.go is a simple starting go file, but can be replaced/changed as needed. It was used
  to verify that the IDE was primed for Go development.
- resources/https-server.crt is a TLS certificate to use for web serving
- resources/https-server.key is a TLS encryption key file to use for web server
- resources/idtrack.js is a version of this app that was written for a different backend, but it expresses
  the desired logic and UX well. The idtrack.html and idtrack.css files in the resource/ directory go with
  this prototype of the web app.

### Overall Architecture

There will be an idtrack Go application that acts as the HTTPS server and as a utility command for affecting
aspects of the tracking system.

When run as a server (based on command line options), it will listen on a port (specified by command line option)
for connections. When connected, it will serve the idtrack.html file (and subsequently the idtrack.css and idtrack.js
files). The web browser that receives these assets will act as the application platform for the webapp.

The webapp will form REST api calls back to the server with endpoints that reflect the abstraction points between the
webapp and the server. currently, the webapp prototype is written to assume a raw SQL interface, so much of the logic
of joining tables, etc. is done in the webapp. Additionally, the current webapp assumes the use of an Ego web server
for authentication and token management required by the Ego endpoints.

This updated version of idtrack will have a simpler interface to the backend, such that the SQL database handling
will all be done on the backend server based on the endpoints requested. So where there is currently a JOIN to combine
issue detail items with issue comments, the new version will just request "an issue" and the SQL operations will all
be done on the backend.

The database will be a SQlite3 database. The actual database file is specified on the command line.

The prototype acquires a bearer token from the backend server, and then the webapp does a simple authentication
check against the back-end. This updated version should have a user log in with username and password, hash the
password, and then use the username and hashed password as Basic authentication to the idtrack backend server,
which will verify each operation against a local database.

### IDTRACK command line

The following command line operations should be supported. The first argument is a verb, followed by options.

- idtrack serve [--port n] [--database path]
- idtrack user --add username:password [--database path]
- idtrack user --delete username:password [--database path]

#### IDTRACK SERVE

This launches idtrack as an HTTPS server. If the port or database are specified on the command line then use
those values, otherwise get defaults from the ~/.idtrack/defaults.json file. The server listens on an
endpoint /idtrack and when found serves the html file. It should also support endpoints for "/assets/idtrack.css"
and "/assets/idtrack.js" which are all read from the resources file.

The server should use the server.crt and server.key files found in resources if present when establishing the
HTTPS listener.

In addition, generate whatever extra endpoints are required by the design of the webapp to serve additional
functionality, such as:

- validate username and password combinations
- get list of issues, with filter and sorting and paging functionality as needed by the UX
- get detail information about an issue
- update an issue
- delete an issue

#### IDTRACK USER

This set of subcommands runs immediately, and updates the Sqlite database specified by the ]--database path]
option. The --add and --delete options will create or delete user entities. Store the password in a hashed value,
ideally matching the hash generated by the Webapp for validating a username and password.

## PROMPT 2

Let's expand the username a little bit. The login and password that exist now are fine. However, let's add the
ability to update the "Display Name" from the user interface.

Add a new option --name "text" to the "user --add" subcommand that specifies a value to be stored as the display
name.

Let's also add a new option "user --update" which accepts a username parameter and then optional additional
flags "--password text" and "--name text" that update the display_name and/or password_hash values in the
users table. The "--update" option only works on an existing name, it cannot be used to create a new name.

## PROMPT 3

Lets add a "--list" option to the "users" verb that lists the users. Do not show the password hash. Also, add
functionality to the webapp that records in the users database when the user last logged in via the webapp's login
operation. (not last time authenticated, as that is each request). Keep this in the users database, and make
it part of --list output.

## PROMPT 4

Let's make it easier to start the server and stop it. When the "serve" verb is issued, let's launch a copy of
idtrack to run the background so the user's terminal session isn't blocked. Record the process id information
of the server, so a "stop" subcommand can be added that stops the server that was started. You can consult the "Ego"
code in ../ego for ideas if that helps, though that is considerably more complex that perhaps what is needed
here.

## PROMPT 5

Let's add a "default" verb that accepts --port and --database options, and writes them to the default.json
file stored in ~/.idtrack such that a user can set them with the "idtrack default" command and then not
have to specify them again when starting the server.

## PROMPT 7

Add an "admin" attribute to user data. If a user has "admin" attribute, they are allowed to DELETE an issue, or
DELETE a comment on an issue. When a user is logged in who has the "admin" attribute, add a "Delete" button to the
top of the issue detail (to the far right of the issue title). Also, add a "trash can" icon next to any existing
comments that the user is allowed to delete. When a delete is requested, confirm with a dialog as this cannot be undone.

For users without the "admin" attribute, do not show these icons or permit the actual delete operations on the
endpoints you'll add to support deleting and issue or a comment.

Add a "--admin true/false" flag to the "idtrack user" command set for adding a new user or updating a user to set
or clear this attribute for a specifc user.
